# Published quants — `glenic/*`

18 models self-quantized (llm-compressor) and published to
`huggingface.co/glenic`, each gated on GSM8K accuracy vs its own bf16 baseline
before upload. `fastserve serve <original-id>` picks these up automatically.
Format is chosen per architecture: **AWQ 4-bit** (W4A16) for models
llm-compressor's AWQ mappings resolve, **W8A8-INT8** for the rest (on A100 the
faster format — INT8 tensor cores, no FP8 — and mapping-free, so it's robust to
new multimodal/MoE architectures). Generated by `publish/batch.py` /
`publish/batch_frontier.py` + retries.

All within the 10pp accuracy gate; none degenerate (<30% threshold). Small
deltas either way are n=30 noise. `*` = the bf16 baseline is understated by a
harness quirk (Yi never emits EOS under plain HF generate; the long-thinker
DeepSeek models truncate at the token budget), not the quant being better —
the point is the quant is *not broken*, which is what the gate proves.

## Frontier models (2026) — mixed formats

Six recent releases, quantized with the format that fits each architecture and
gated on GSM8K (n=15). Gemma-4 are multimodal (vision **+** audio towers on the
larger "omni" variants) and the pipeline keeps those towers — and the MoE
router — at full precision automatically.

| Model (original id) | Format | bf16 baseline | our quant | degenerate | note |
|---|---|---|---|---|---|
| Qwen/Qwen3.6-27B | AWQ 4-bit | 0.800 | 0.800 | 0/15 | dense; AWQ via dynamic mappings |
| google/gemma-4-31B-it | W8A8-INT8 | 0.800 | 0.800 | 0/15 | dense omni (vision+audio) |
| google/gemma-4-26B-A4B-it | W8A8-INT8 | 0.533 | 0.600 | 1/15 | 128-expert MoE; router kept fp |
| google/gemma-4-12B-it | W8A8-INT8 | 0.733 | 0.667 | 0/15 | dense omni (vision+audio) |
| google/gemma-4-E4B-it | W8A8-INT8 | 0.467 | 0.533 | 0/15 | elastic (MatFormer) |
| google/gemma-4-E2B-it | W8A8-INT8 | 0.467 | 0.467 | 0/15 | elastic (MatFormer) |

Two failure modes found and fixed here, now permanent in the pipeline: (1) the
omni models carry an **audio** tower (`embed_audio`) that vLLM needs at full
precision — quantizing it makes the checkpoint unloadable; (2) INT8-ing a
128-expert MoE **router** collapses routing into repeated-token garbage
(GSM8K 0.53 → 0.00) until the router is kept full precision. Both are auto-ignored
now (`re:.*audio.*`, `re:.*router.*`).

## The headline result — the two Gemma quants

fastserve's own benchmark caught the community AWQ quants of both Gemma-2
models producing **degenerate garbage** (looped tokens, GSM8K accuracy
**0.000**; see `../benchmarks/RESULTS.md`). Our self-quantized replacements
answer correctly (0.467 / 0.633, non-degenerate) and `detect()` now prefers
them:

```
$ fastserve info google/gemma-2-2b-it
  found pre-quantized checkpoint: glenic/gemma-2-2b-it-AWQ (AWQ)   # not the broken community one
```

Gemma needed `--calib-dataset wikitext` because its chat template rejects the
`system`-role messages in ultrachat-200k (`jinja2 TemplateError: System role
not supported`).

## Not published (and why)

Multimodal models (Qwen3.5/3.6, gemma-4) are now **handled** — the pipeline
loads them via `AutoModelForImageTextToText`, keeps the vision/audio towers and
MoE router at full precision, and saves the full multimodal config so vLLM
loads the result. What still can't be done on this hardware:

| Model | Why not |
|---|---|
| Kimi-Linear-48B-A3B | custom tokenizer import error + 256-expert MoE + linear attention — the pipeline can't quantize this combination cleanly |
| Leanstral-1.5-119B-A6B | bf16 ~238 GiB — can't be loaded to quantize on 2×80 GiB, and there's no baseline to gate against |
| gpt-oss-120b / gpt-oss-20b | already ship **native MXFP4** (4-bit out of the box) — re-quantizing gains nothing |
| Qwen2.5-72B and larger dense | bf16 baseline won't fit one 80GB GPU, so there's no accuracy reference to gate against |
| gated Llama | license not accepted on the `glenic` account behind `HF_TOKEN` |
| Phi-3.5-mini | stale custom modeling code (`DynamicCache.seen_tokens`) fails the HF-eager baseline |
